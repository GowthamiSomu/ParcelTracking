# Deployment Guide

Two deployment modes are supported. Both expose identical ports so the demo runner and API clients need no reconfiguration.

| Port | Service |
|------|---------|
| `5058` | REST API + Swagger UI |
| `8080` | Adminer (database browser) |
| `5672` | Azure Service Bus emulator (AMQP) |

## Option A â€” Docker Compose (recommended for local dev/demo)

### Prerequisites
- [Docker Desktop](https://www.docker.com/products/docker-desktop/) â‰Ą 4.x

### Start

```bash
# 1. Create your .env from the template (one-time)
copy .env.example .env
# Edit .env and set a strong SA_PASSWORD

# 2. Start all 7 containers
docker compose up -d

# 3. Verify all containers are healthy
docker compose ps
```

Containers started: `sqlserver`, `redis`, `sqledge` (Service Bus backing store), `servicebus-emulator`, `adminer`, `ingestion`, `api`.

### Useful commands

```bash
# Stream ingestion logs
docker compose logs -f ingestion

# Stop everything
docker compose down

# Stop + wipe all data (volumes)
docker compose down -v
```

## Option B â€” Kubernetes (Docker Desktop)

### Prerequisites
- Docker Desktop with **Kubernetes enabled** (Settings â†’ Kubernetes â†’ Enable)

### Setup (one command)

```powershell
.\deploy\k8s\k8s-demo-setup.ps1
```

The script:
1. Verifies `kubectl` context is `docker-desktop`
2. Builds local images with `docker compose build api ingestion`
3. Creates the K8s secret from your `.env` file (`create-secrets.ps1`)
4. Applies all manifests in order (`00-namespace` â†’ `09-api`)
5. Waits up to 3 minutes for every pod to be Ready
6. Prints live URLs

### Manual manifest apply (if needed)

```powershell
kubectl apply -f deploy/k8s/00-namespace.yaml
# Run create-secrets.ps1 before continuing
.\deploy\k8s\create-secrets.ps1
kubectl apply -f deploy/k8s/02-configmap.yaml
kubectl apply -f deploy/k8s/03-sqlserver.yaml
kubectl apply -f deploy/k8s/04-redis.yaml
kubectl apply -f deploy/k8s/05-sqledge.yaml

kubectl apply -f deploy/k8s/07-adminer.yaml
kubectl apply -f deploy/k8s/08-ingestion.yaml
```

## Secret management

Credentials are **never** stored in source control. The `.env` file (gitignored) is the single source of truth for local deployments.

| Variable | Description |
|---|---|
| `SA_PASSWORD` | SQL Server `sa` account password (â‰Ą 8 chars, upper + lower + digit + symbol) |
| `SQLSERVER_CONNECTION` | Full EF Core connection string |
| `REDIS_CONNECTION` | `host:port` for StackExchange.Redis |
| `SERVICEBUS_CONNECTION` | Azure Service Bus emulator connection string |

Copy `.env.example` to `.env` and fill in values before starting any deployment.
